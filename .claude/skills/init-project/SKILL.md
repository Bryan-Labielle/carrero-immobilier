# Laravel + Inertia + Vue.js Project Setup

This skill configures a freshly installed Laravel project with all code quality tools and best practices. It runs through the following sequential steps.

Before starting, verify that the Laravel project exists in the `src/` subdirectory (presence of `src/composer.json`, `src/artisan`, etc.). If not, ask the user to create it with `laravel new src` or rename their project directory to `src/`.

All file paths in this skill are relative to `src/` unless stated otherwise.

## Steps Overview

1. Node.js and PHP versions
2. PHP Pint (PHP linter)
3. ESLint (JS/TS linter)
4. Prettier (code formatter)
5. Pre-commit with Husky + lint-staged
6. Laravel AppServiceProvider
7. Install CLAUDE.md rules
8. Global linting
9. Dependency updates
10. Clean git history & initial commit

## STEP 1: Node.js and PHP Versions
é
### 1.1 - `.nvmrc` file
Create the `.nvmrc` file at the project root (same level as `composer.json`). It should contain only the current LTS major version number of Node.js (e.g., `22`), with no prefix or suffix.

### 1.2 - PHP version in `composer.json`
In the `"require"` section of `composer.json`, update the PHP constraint to match the version specified in CLAUDE.md. Format: `"php": "^X.Y"` (the `^` allows compatible minor updates).

## STEP 5: Pre-commit with Husky + lint-staged

This step requires steps 2, 3, and 4 to be completed.

### 5.1 - Installation

```bash
npm install --save-dev husky lint-staged
```

Husky will add a `"prepare"` script to `package.json`. Do NOT run `npx husky init` — the `.git` directory is in the parent, not in `src/`, so it will fail.

### 5.2 - Create `.husky/` directory and pre-commit hook

Create the directory and file manually:

```bash
mkdir -p .husky
```

Create `.husky/pre-commit` with this content:

```bash
cd src
npx lint-staged
```

Make it executable:

```bash
chmod +x .husky/pre-commit
```

The `cd src` is necessary because the hook runs from the Git repository root, not from the Laravel directory.

### 5.3 - Prepare script

The Laravel project lives in a `src/` subdirectory of the Git repository. Update the `"prepare"` script in `package.json`:

```json
"prepare": "cd .. && husky src/.husky || true"
```

The `|| true` prevents errors if Husky is not yet installed (e.g., in CI).

Then run it to register the hooks:

```bash
npm run prepare
```

### 5.4 - lint-staged configuration

Add the `"lint-staged"` object at the root level of `package.json` (same level as `"scripts"`):

```json
"lint-staged": {
    "*.php": [
        "./vendor/bin/pint --config pint.json"
    ],
    "*.{js,ts,vue}": [
        "prettier --write",
        "eslint --fix"
    ]
}
```

### 5.5 - Test the configuration

1. Intentionally introduce formatting errors in a PHP or Vue file (extra spaces, excessive line breaks)
2. Stage and commit:
   ```bash
   git add .
   git commit -m "Test pre-commit hook"
   ```
3. Verify the hook triggers and fixes the errors
4. If the test succeeds, undo the test commit:
   ```bash
   git reset --soft HEAD~1
   ```
5. If the hook does not trigger, check permissions: `chmod +x .husky/pre-commit`

## STEP 6: AppServiceProvider

### 6.1 - boot() method and private methods

In `app/Providers/AppServiceProvider.php`, replace the `boot()` method with the following code. Each responsibility is extracted into a private method to keep `boot()` readable:

```php
public function boot(): void
{
    $this->configureModels();
    $this->configurePasswordReset();
    $this->configureRoutes();
    $this->configureUrls();
    $this->configureMailRouting();
}

private function configureModels(): void
{
    Model::preventLazyLoading(!$this->app->isProduction());
    Model::preventSilentlyDiscardingAttributes(!$this->app->isProduction());
}

private function configurePasswordReset(): void
{
    ResetPassword::toMailUsing(function ($notifiable, $token) {
        $url = url(route('password.reset', [
            'token' => $token,
            'email' => $notifiable->getEmailForPasswordReset(),
        ], false));

        return (new MailMessage())
            ->subject('Reinitialisation de votre mot de passe')
            ->line('Vous recevez cet e-mail car nous avons recu une demande de reinitialisation de mot de passe pour votre compte.')
            ->action('Reinitialiser le mot de passe', $url)
            ->line('Ce lien de reinitialisation expirera dans ' . config('auth.passwords.' . config('auth.defaults.passwords') . '.expire') . ' minutes.')
            ->line('Si vous n\'avez pas demande de reinitialisation, aucune autre action n\'est requise.');
    });
}

private function configureRoutes(): void
{
    Route::resourceVerbs([
        'create' => 'creer',
        'edit' => 'modifier',
    ]);
}

private function configureUrls(): void
{
    URL::forceHttps($this->app->environment(['production', 'develop']));
}

private function configureMailRouting(): void
{
    $alwaysTo = match ($this->app->environment()) {
        'local' => config('mail.to.developer'),
        'develop' => config('mail.to.support'),
        default => null,
    };

    if ($alwaysTo) {
        Mail::alwaysTo($alwaysTo);
    }
}
```

Keep any existing methods already present in the file (like `configureDefaults()`). Add the new methods alongside them, and call them from `boot()`.

### 6.2 - Imports

Add the necessary imports at the top of the file (no duplicates):

```php
use Illuminate\Database\Eloquent\Model;
use Illuminate\Auth\Notifications\ResetPassword;
use Illuminate\Notifications\Messages\MailMessage;
use Illuminate\Support\Facades\Route;
use Illuminate\Support\Facades\URL;
use Illuminate\Support\Facades\App;
use Illuminate\Support\Facades\Mail;
```

### 6.3 - Email configuration

In `config/mail.php`, add after the `'from'` key:

```php
'to' => [
    'developer' => env('MAIL_TO_DEVELOPER', 'support-technique@agence-awam.fr'),
    'support' => env('MAIL_TO_SUPPORT', 'support-technique@agence-awam.fr'),
],
```

In `.env`, add:
```env
MAIL_TO_DEVELOPER=votre.email.dev@exemple.com
MAIL_TO_SUPPORT=support-technique@agence-awam.fr
```

In `.env.example`, add:
```env
MAIL_TO_DEVELOPER=dev@example.com
MAIL_TO_SUPPORT=support@example.com
```

### 6.4 - Storage symlink

```bash
php artisan storage:link
```

### 6.5 - Locale and timezone configuration

In both `.env` and `.env.example`, update the following variables:

```env
APP_LOCALE=fr
APP_FALLBACK_LOCALE=fr
APP_FAKER_LOCALE=fr_FR
APP_TIMEZONE=Europe/Paris
```

## STEP 6.6: HandleInertiaRequests

In `app/Http/Middleware/HandleInertiaRequests.php`, add the following line to the `share()` method's return array:

```php
'query' => $request->query(),
```

This shares the current query parameters with all Inertia pages, making them accessible on the client side.

## STEP 7: Install CLAUDE.md Rules

Copy the CLAUDE.md files from the toolbox repo root into the Laravel project, then clean up.

Create missing directories first:

```bash
mkdir -p resources/js/{Components,Composables,Pages,Stores}
```

Copy each CLAUDE.md into the matching project directory:

```bash
cp ../../resources/js/CLAUDE.md resources/js/CLAUDE.md
cp ../../resources/js/Components/CLAUDE.md resources/js/Components/CLAUDE.md
cp ../../resources/js/Composables/CLAUDE.md resources/js/Composables/CLAUDE.md
cp ../../resources/js/Pages/CLAUDE.md resources/js/Pages/CLAUDE.md
cp ../../resources/js/Stores/CLAUDE.md resources/js/Stores/CLAUDE.md
```

Then remove the toolbox's `resources/` directory so it doesn't pollute the repo:

```bash
rm -rf ../../resources/
```

## STEP 10: Clean Git History

This step removes all commits inherited from the laravel-starter-kit repo, so the new project starts with a fresh history.

### 10.1 - Ask for the new remote URL

Ask the user for their new Git remote URL (e.g., `git@github.com:user/new-project.git`). If they don't have one yet, skip the remote step — they can add it later.

### 10.2 - Create orphan branch

```bash
cd /path/to/repo-root   # the parent of src/, where .git lives
git checkout --orphan main-clean
git add -A
git commit -m "Initial commit"
```

### 10.3 - Replace main branch

```bash
git branch -D main
git branch -m main
```

If the project also used a `develop` branch, delete it:

```bash
git branch -D develop 2>/dev/null || true
```

### 10.4 - Update remote

If the user provided a new remote URL:

```bash
git remote set-url origin <new-remote-url>
```

If the user wants to remove the old remote entirely first:

```bash
git remote remove origin
git remote add origin <new-remote-url>
```

### 10.5 - Verification

```bash
git log --oneline -5
git status
git remote -v
```

Expected result:
- `git log` shows a single "Initial commit"
- `git status` shows "working tree clean"
- `git remote -v` shows the new remote (or no remote if skipped)
